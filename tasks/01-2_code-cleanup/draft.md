## T01 [P0] デバッグ用ログや仮実装を整理する

- タスク内容:
  - デバッグ用ログや仮実装を整理する
- 目的:
  - 公開版ビルドで不要ログと暫定コードを残さないため。
- 草案:
  - 削除対象: `DEBUG`/`TODO`付きの一時表示、強制分岐用フラグ、検証専用定数。
  - 残す対象: 不具合調査に必要な最小エラーログ（致命エラー・設定読込失敗）。
  - 成果物: `削除項目一覧(ファイル/理由)` と `残置ログ一覧(用途/出力条件)` を1表にまとめる。
  - 注意点: 削除後に差分レビューし、走行ロジックに影響する削除を防ぐ。
- 結果:
  - `公開版ビルドで常時出力されていた設定ログを停止し、診断ログはすべてフラグ無効で残す方針に整理する。`

## T02 [P0] 不要な設定や未使用コードを削除する

- タスク内容:
  - 不要な設定や未使用コードを削除する
- 目的:
  - 保守対象を減らし、公開後の不具合混入を抑えるため。
- 草案:
  - 削除対象: 参照されない設定キー、呼び出し元のない関数、未使用リソース。
  - 確認手順: 検索結果と参照元を一覧化し、誤削除防止のためレビューで承認を取る。
  - 成果物: `削除前後の一覧` と `影響なし確認メモ`。
  - 注意点: 公開範囲外コードは「残す理由」が無い限り削除対象とする。
- 結果:
  - 削除前後の一覧:
    - 削除前: `MarathonCoachField` に、定数で常時無効化されていたカードプレビュー分岐とレイアウトデバッグ描画が残っていた。
    - 削除前: `MarathonCoachField._containsNonAscii()` は呼び出し元のない単純ラッパーだった。
    - 削除後: 本番経路に一切入らない `_applyCardVariantPreview()` と関連定数、レイアウトデバッグ描画、未使用ラッパーを削除した。
  - 影響なし確認メモ:
    - 通常のカード切り替え・描画・通知ロジックには未到達コードのみを削除しており、機能仕様は変更していない。
    - 利用者価値: 読み解くべき分岐が減り、将来の修正時に誤ってプレビュー専用経路へ触れるリスクを下げる。
    - 起動速度/安定性への影響メモ: 劇的な体感差は想定しないが、初期読込対象と描画分岐のノイズを減らし、保守時の安全性を上げる。

## T03 [P1] バージョン番号の運用ルールを決める

- タスク内容:
  - バージョン番号の運用ルールを決める
- 目的:
  - 公開後の更新内容を利用者に一貫して伝えるため。
- 草案:
  - 採用ルール: Semantic Versioning（`MAJOR.MINOR.PATCH`）。
  - 初回公開: `1.0.0`、不具合修正は `PATCH`、軽微改善は `MINOR`、互換性変更は `MAJOR`。
  - β運用: 正式公開前は `0.x.y`、外部公開する版のみタグを付ける。
  - 注意点: Store表示バージョンとGitタグを一致させる。
- 結果:
  - `Semantic Versioning（MAJOR.MINOR.PATCH）` を採用する。
  - 初回公開は `1.0.0`、不具合修正は `PATCH`、軽微改善は `MINOR`、互換性変更は `MAJOR` とする。
  - 正式公開前のβ運用は `0.x.y` を使い、外部公開する版のみ Git タグを付ける。
  - Store表示バージョンとGitタグは常に一致させる。

## T04 [P1] 更新履歴テンプレートを作る

- タスク内容:
  - 更新履歴テンプレートを作る
- 目的:
  - 更新内容を短時間で作成し、説明漏れを防ぐため。
- 草案:
  - テンプレート見出し: `追加` / `変更` / `修正` / `既知の制限`。
  - 記載例: `修正: 補給通知が連続発火する不具合を修正（FR255で再現）`。
  - 公開先: Store説明文、X投稿、RELEASEノートで同一文面を使用。
  - 注意点: 内部事情ではなく、利用者が受ける影響を主語に書く。
- 結果:
  - テンプレート見出しは `追加 / 変更 / 修正 / 既知の制限 / ランナーへの影響` とする。
  - 記載例は `修正: 補給通知が連続発火する不具合を修正（FR255で再現）` を採用する。
  - Store説明文、X投稿、RELEASEノートで同一文面を使い回す運用とする。
  - 記述は内部事情ではなく、利用者が受ける影響を主語に統一する。
