## T01 [P0] 通常版の提供範囲を1文で定義する

- タスク内容:
  - 通常版の提供範囲を1文で定義する
- 目的:
  - 通常版の価値を一文で固定し、仕様判断の基準を統一するため。
- 草案:
  - 提供範囲の確定文: `通常版は、目標タイム連動の表示と補給通知で、レース中の判断負荷を下げる1画面データフィールドを提供する。`
  - 含める要素: 目標タイム連動 / ACTION表示 / 35分固定補給通知 / ビープ通知。
  - 含めない要素: 個別分析レポート、個別調整、有人サポート付き設定最適化。
  - 注意点: 一文に価値・対象・提供物を含め、解釈の余地を残さない。

## T02 [P0] カスタム版の提供範囲を1文で定義する

- タスク内容:
  - カスタム版の提供範囲を1文で定義する
- 目的:
  - 有料サービスの価値を一文で固定し、通常版との棲み分けを明確にするため。
- 草案:
  - 提供範囲の確定文: `カスタム版は、提出されたFITデータとヒアリング回答を基に、個人専用の設定値・根拠説明・カスタムコードを納品する。`
  - 納品内容に必ず含める: 推奨設定値 / 指標根拠 / 入力手順 / 再調整条件。
  - 通常版との境界: アプリ本体の無料利用と、個別分析レポートは分離する。
  - 注意点: 「分析する」ではなく「何を納品するか」まで明記する。

## T03 [P0] 無料部分と有料部分の境界を確定する

- タスク内容:
  - 無料部分と有料部分の境界を確定する
- 目的:
  - 料金説明と機能説明の不一致を防ぐため。
- 草案:
  - 無料(通常版): 目標タイム入力 / 距離設定 / レース中表示 / 35分補給通知 / ビープ通知。
  - 有料(カスタム版): FIT分析レポート / 7項目評価 / 個別設定値 / カスタムコード / 再調整1回。
  - 境界ルール: 「個人データの解釈と個別提案」が発生する作業は有料扱い。
  - 例外: 不具合修正、公開済み通常版のバグ対応は無料。
  - 注意点: 無料・有料の両方に具体例を入れて問い合わせ時の説明コストを下げる。

## T04 [P0] 対象ユーザー像を決める

- タスク内容:
  - 対象ユーザー像を決める
- 目的:
  - 誰に最適化するかを固定し、機能追加の判断軸を明確にするため。
- 草案:
  - 主対象: `フルマラソン完走〜サブ4を狙う市民ランナー`。
  - 課題像: レース中にペース判断・補給判断で迷い、後半失速しやすい。
  - 対象外: トレイル専用運用、医療用途、競歩・自転車向け利用。
  - 訴求軸: 迷い削減 / 補給忘れ防止 / 失速兆候への先回り。
  - 注意点: 対象外を先に明記して、過剰な期待値を防ぐ。

## T05 [P0] 公開時の最小完成条件を決める

- タスク内容:
  - 公開時の最小完成条件を決める
- 目的:
  - 公開延期判断を迷わず実施できる基準を作るため。
- 草案:
  - 必須条件: P0タスク完了 / 致命的不具合0件 / 初回設定で利用開始可能。
  - 証跡条件: 主要機種帯の実機確認結果、ストア素材、サポート窓口文面が揃っている。
  - 延期条件: 走行中誤誘導の可能性がある通知不具合、設定保存不可、クラッシュ再現。
  - 注意点: 「できれば」を排除し、公開可否を二択で判定できる形式にする。
